## Threat Actors

### External Attacker (Internet-facing)
- No prior access to systems
- Motivated by: financial gain (drain validator stake, bypass payment), disruption, data theft
- Capabilities: HTTP-level attacks, automated scanning, public blockchain analysis

### Malicious Insider
- Has legitimate access to development or operations systems
- Motivated by: financial gain, sabotage, espionage
- Capabilities: direct database access, ability to modify environment variables, access to private keys stored insecurely

### Compromised Validator
- A compute validator in the `membra_attestation` network that has been bribed or compromised
- Motivated by: gaming the reward system, harming competing projects
- Capabilities: submit attestations, participate in challenge process, observe all on-chain project data

### Supply Chain Attacker
- Targets npm, PyPI, or crates.io dependencies
- Motivated by: widespread code execution in many deployments
- Capabilities: inject malicious code into a dependency used by this project

## STRIDE Analysis

---

### Component: React Frontend

#### Spoofing — Wallet Impersonation
- **Description:** An attacker constructs a page that mimics the MEMBRA frontend and tricks a user into signing a malicious transaction that drains their wallet.
- **Likelihood:** MEDIUM (phishing is common in Web3)
- **Impact:** HIGH (user funds drained)
- **Current Mitigations:** The legitimate domain is shown in the wallet popup; users are trained to verify the origin domain before signing. `Strict-Transport-Security` prevents downgrade attacks.
- **Recommended:** Add a `manifest.json` with canonical origin; display the artifact PDA address on the signing prompt for cross-verification.

#### Tampering — QR Redirect Manipulation
- **Description:** An attacker generates a QR code pointing to a malicious URL that looks like a MEMBRA artifact page. The user scans it and is tricked into signing a fraudulent transaction.
- **Likelihood:** MEDIUM
- **Impact:** HIGH
- **Current Mitigations:** QR codes generated by the API encode the `artifact_id`; the destination page fetches artifact metadata from the API and verifies the `artifact_hash` before presenting transaction details. Hash mismatch surfaces an error.
- **Recommended:** Display the full `destination_url` prominently before any signing prompt; add a browser extension warning for known phishing domains.

#### Repudiation — Unsigned Actions
- **Description:** A user claims they did not perform an action (e.g., registering an artifact) because no cryptographic proof of their identity exists for that action.
- **Likelihood:** LOW
- **Impact:** MEDIUM
- **Current Mitigations:** Wallet-authenticated sessions tie actions to Ed25519 signatures. Audit log records actor and timestamp for all state-changing operations.
- **Recommended:** Require wallet signature for artifact registration (not just API key) to establish non-repudiation for high-value artifacts.

#### Information Disclosure — Wallet Address Exposure
- **Description:** The frontend logs wallet addresses to the browser console or includes them in error reports, exposing user pseudonymity.
- **Likelihood:** LOW (fixed in v1.1.0)
- **Impact:** MEDIUM
- **Current Mitigations:** `console.log` of wallet address removed in v1.1.0. Wallet address is displayed intentionally in the UI per user consent.
- **Recommended:** Automated lint rule to ban `console.log` containing `publicKey` or `address`.

#### Denial of Service — Heavy QR Generation
- **Description:** An attacker submits many POST `/api/qr` requests to exhaust server resources or fill the SQLite database.
- **Likelihood:** MEDIUM
- **Impact:** MEDIUM (service degradation)
- **Current Mitigations:** Rate limiting (100 req/min per IP). API key required for QR registration.
- **Recommended:** Add Captcha or proof-of-work for unauthenticated QR generation; monitor artifact count growth.

#### Elevation of Privilege — Admin Bypass via IDOR
- **Description:** A user discovers admin endpoints and sends requests with a guessed or leaked `ADMIN_API_KEY`.
- **Likelihood:** LOW
- **Impact:** CRITICAL
- **Current Mitigations:** `ADMIN_API_KEY` is validated via constant-time comparison; admin endpoints return 403 if key is absent or wrong.
- **Recommended:** Restrict admin endpoint access to an internal-only network or VPN; add IP allowlist at the Nginx level.

### Component: FastAPI API

#### Injection — SQL Injection
- **Description:** An attacker sends crafted input that modifies SQL queries, enabling data exfiltration or modification.
- **Likelihood:** LOW
- **Impact:** CRITICAL
- **Current Mitigations:** All queries use parameterized statements (`aiosqlite` with `?` placeholders). Pydantic input validation rejects unexpected characters. `sanitise_sql_like` escapes LIKE clause wildcards.
- **Recommended:** Run `sqlfluff` or similar static analysis to verify no raw string interpolation in queries.

#### Authentication Bypass — JWT Algorithm Confusion
- **Description:** An attacker submits a JWT signed with the `none` algorithm or RS256 when the server expects HS256.
- **Likelihood:** LOW
- **Impact:** HIGH
- **Current Mitigations:** `jwt.decode` is called with `algorithms=["HS256"]` explicitly; `none` algorithm is rejected by `python-jose`.
- **Recommended:** Add an integration test that confirms a `none`-algorithm JWT is rejected with 401.

#### Server-Side Request Forgery — Webhook URL
- **Description:** An artifact is registered with a `destination_url` pointing to an internal service (e.g., `http://169.254.169.254/` for cloud metadata), and the API fetches it during processing.
- **Likelihood:** LOW
- **Impact:** HIGH
- **Current Mitigations:** `destination_url` is validated to require `http` or `https` scheme. The API does not currently fetch `destination_url` server-side; it is stored and returned to the client for redirect.
- **Recommended:** Add an SSRF allowlist/denylist for any future server-side URL fetching; block RFC 1918 and link-local ranges.

#### Rate Limit Bypass — Multiple IPs / Distributed Attack
- **Description:** An attacker distributes requests across many IP addresses to bypass per-IP rate limiting.
- **Likelihood:** MEDIUM
- **Impact:** MEDIUM
- **Current Mitigations:** Per-IP token bucket in-process. API key is required for most mutation endpoints.
- **Recommended:** Add per-API-key rate limiting as a secondary layer; use Nginx `limit_req_zone` by `$binary_remote_addr`.

#### Information Disclosure — Stack Traces in Error Responses
- **Description:** Unhandled exceptions leak internal stack traces or database schema information to the caller.
- **Likelihood:** LOW (handled in v1.1.0)
- **Impact:** MEDIUM
- **Current Mitigations:** `unhandled_exception_handler` in `core/errors.py` returns a generic "Internal server error" message; exception details are logged server-side only.
- **Recommended:** Add integration tests asserting that all error responses contain only `{ok, error, detail}` fields and no stack traces.

### Component: Solana Programs

#### Reentrancy
- **Description:** A cross-program invocation (CPI) callback is triggered before state is finalized, enabling a classic reentrancy attack.
- **Likelihood:** LOW (Solana's execution model differs from EVM)
- **Impact:** HIGH if exploitable
- **Current Mitigations:** Solana's account model requires that all accounts be declared upfront; the runtime does not permit reentrant CPI to the same program in the same transaction. Anchor enforces account mutability constraints.
- **Recommended:** N/A for standard Solana programs; document this explicitly for auditors.

#### Authority Spoofing — Unauthorized Instruction Execution
- **Description:** An attacker calls a privileged instruction (e.g., `publish_project_score`, `resolve_challenge`) without being the configured `authority`.
- **Likelihood:** LOW
- **Impact:** CRITICAL
- **Current Mitigations:** All privileged instructions use `constraint = authority.key() == config.authority` guards enforced by Anchor at the account validation layer. Instruction handler is never reached if the constraint fails.
- **Recommended:** Ensure authority is checked against a PDA-stored key rather than a signer key wherever possible; this prevents authority key compromise from being the single point of failure.

#### PDA Collision — Deterministic Seed Attacks
- **Description:** An attacker crafts inputs that produce the same Program Derived Address (PDA) as a legitimate account, overwriting or front-running it.
- **Likelihood:** LOW
- **Impact:** HIGH
- **Current Mitigations:** PDAs use typed, high-entropy seeds (e.g., `b"attestation_config"`, `validator.key()`, `project_id` as `u64`). Anchor `init` instruction fails if the PDA already exists.
- **Recommended:** Include the program ID in PDA seeds where cross-program seed conflicts are possible; audit seed uniqueness in `membra_attestation` project PDAs.

#### Arithmetic Overflow — Token Amount Calculations
- **Description:** Integer overflow in token amount or score arithmetic produces incorrect results or panics.
- **Likelihood:** LOW
- **Impact:** HIGH
- **Current Mitigations:** Rust's default debug mode panics on overflow. Release mode uses `checked_add`, `checked_mul`, `saturating_*` operations throughout reward and rebase calculations.
- **Recommended:** Enable `overflow-checks = true` in `[profile.release]` in `Cargo.toml` for all program crates.

#### Front-running — Attestation Score Manipulation
- **Description:** A validator observes a pending attestation transaction in the mempool and submits their own attestation immediately before it to influence the running average.
- **Likelihood:** MEDIUM (Solana has lower mempool exposure than EVM but leader-based ordering exists)
- **Impact:** MEDIUM
- **Current Mitigations:** Attestation scores are stake-weighted; a single validator cannot disproportionately influence the average without controlling a large stake. Challenge mechanism allows disputed attestations to be removed.
- **Recommended:** Consider a commit-reveal scheme for attestation scores to prevent observable front-running.

#### Validator Collusion — Coordinated Fake Attestations
- **Description:** A group of validators collude to submit false attestations for a project, pushing its risk score artificially high or low.
- **Likelihood:** LOW-MEDIUM
- **Impact:** HIGH (market manipulation)
- **Current Mitigations:** Challenge mechanism allows any party to dispute an attestation. Upheld challenges slash validator stake. Reputation tracking (`completed_jobs`, `slash_count`) provides long-term accountability signal.
- **Recommended:** Set `min_attestations` high enough that collusion requires a majority of staked validators; implement a minimum stake requirement that makes collusion economically costly.
